Claude Code Policy for Junior Developers

Your Quick Reference Guide

LEARNING & DEVELOPMENT

Welcome to the Team!

This simplified guide explains how to use Claude Code as a junior developer. For the complete policy, see the full Internal Policy document.

✅ What You CAN Do

1. Write Features With Claude Help

You can ask Claude to help you build features:

  • Ask Claude for code structure/architecture
  • Ask Claude to generate code based on requirements
  • Ask Claude to explain how features work
  • Ask Claude to help with specific patterns you're learning

2. Debug And Fix Problems

  • Share error messages and ask for help
  • Share code snippets (without secrets)
  • Ask Claude to explain why something isn't working
  • Ask for refactoring suggestions

3. Write Tests

  • Ask Claude to generate test cases
  • Ask for testing patterns and best practices
  • Ask how to test edge cases

4. Learn Programming Concepts

  • Ask Claude to explain React hooks, async/await, etc.
  • Ask for examples of design patterns
  • Ask how to structure code following best practices

🏆 The Golden Rules (NEVER Break These)

Rule 1: NEVER Share Secrets

Secrets = API keys, passwords, tokens, private keys

What NOT to do:

❌ "Here's my .env file: STRIPE_KEY=sk_live_xyz"

What TO do:

✅ "Why isn't my Stripe integration working?" (describe problem, not the secret)

If you accidentally share a secret:

  1. Tell your manager immediately
  2. Tell the security team
  3. Rotate the secret right away

Rule 2: NEVER Share Production Data

Don't share: Real customer emails, payment info, database dumps

❌ SELECT * FROM users WHERE id=45328; Results: John Smith, john@gmail.com, credit card 4532-1234... ✅ "How do I optimize this query?" (describe structure, not actual data)

Rule 3: NEVER Commit Code You Don't Understand

If Claude generates code and you don't understand it:

  1. Ask Claude to explain each part
  2. Ask your senior to review it first
  3. Don't commit until you understand it completely

Remember: YOU are responsible for code you commit, even if Claude generated it.

Rule 4: NEVER Skip Testing

Before you make a PR:

  • ✓ Run linter: npm run lint
  • ✓ Run tests: npm test
  • ✓ Test manually: Does the feature actually work?
  • ✓ Ask: "Did I test the error cases?"

Rule 5: NEVER Merge Your Own Code

Always ask a senior to review before merging:

  1. Create PR with clear description
  2. Request review from senior developer
  3. Wait for approval
  4. Senior merges (not you)

📋 Your Workflow (Step By Step)

Step 1: Before You Ask Claude

  • Read the feature requirements carefully
  • Check if similar code already exists in our codebase
  • Ask your mentor/senior if you're not sure

Step 2: Ask Claude (Safely)

Good prompt:

"Create a React component for user login with: - Email and password fields - Validation (email format, min 8 char password) - Submit to POST /api/auth/login - Show loading state - Handle errors gracefully - No storing password in state - Use TypeScript"

Bad prompt:

"Create login component" ← Too vague!

Step 3: Review Code Locally

  • Read every line of generated code
  • Ask Claude to explain anything you don't understand
  • Check if it matches team conventions

Step 4: Test Everything

  • npm run lint - Check for style issues
  • npm test - Run tests
  • npm run build - Verify it builds
  • Manual test - Does the feature actually work?

Step 5: Ask Senior to Review Before PR

For AI-generated code, have a senior review locally first (not in PR).

Ask: "Does this code look good? Any security issues?"

Step 6: Create PR & Request Review

  • Describe what the code does
  • Link to the GitHub issue/ticket
  • Note if code was AI-assisted: "⚠️ Generated with Claude Code"
  • Request review from senior developer

Step 7: Respond To Review Comments

  • Read every comment carefully
  • Ask questions if you don't understand
  • Make changes requested
  • Push updates and request re-review

Step 8: Senior Merges

Once approved, your senior merges the PR. You don't merge your own code.

❌ Common Mistakes (Learn From Others)

Mistake 1: Blind Trust

What happened: Junior asked Claude for auth code, copied it directly without understanding, code had security bug.

What TO do: Always understand code before using it. If you don't understand, ask questions.

Mistake 2: Sharing Secrets

What happened: Junior pasted .env file with API keys to Claude. Keys were compromised.

What TO do: Never paste actual secrets. Use placeholder values instead.

Mistake 3: No Testing

What happened: Junior skipped tests, deployed code, feature broke in production.

What TO do: Always run tests before PR. Always test manually.

Mistake 4: Merging Own Code

What happened: Junior approved their own PR and merged without review. Bug went to production.

What TO do: Never approve your own code. Always request senior review.

🤝 When To Ask For Help (It's OK!)

You should ask a senior developer when:

  • You don't understand Claude's code
  • You're not sure if code is correct
  • You don't know how to test something
  • You're making security-related code
  • You need to debug a complex issue
  • You're learning a new pattern/concept
  • Anything where you feel unsure

Remember: Asking for help is a sign of a good developer, not weakness. We want you to learn and grow!
