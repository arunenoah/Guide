1. Introduction & Overview

Purpose

This document is the single source of truth for all developers using Claude Code in our organization. It covers:

  • Security requirements and threat mitigation
  • Proper use of Claude Code and custom subagents
  • Development workflow and code review process
  • Deployment procedures and incident response
  • Best practices for code quality and maintainability

Who This Applies To

All developers - juniors, mid-level, and seniors. There are no role-based exceptions. The security requirements apply uniformly to everyone.

Core Principles

  • Defense in Depth: Multiple security layers protect against single failures
  • Zero Trust: Verify everything; assume nothing is secure by default
  • Least Privilege: Subagents and tools restricted to minimum necessary access
  • Audit Everything: All Claude Code usage, agent creation, and deployments logged
  • Fail Safe: Code never commits/deploys without human review

2. Critical Non-Negotiable Rules

Violations of these rules result in immediate suspension of Claude Code access.

RULE 1: NEVER SHARE SECRETS WITH CLAUDE No API keys, passwords, tokens, database credentials, or sensitive configuration values. Ever. Use placeholder values: sk_test_xxx, YOUR_API_KEY_HERE, password_placeholder
RULE 2: NEVER SHARE PRODUCTION DATA WITH CLAUDE No customer records, personally identifiable information (PII), database dumps, transaction data, or any real user information. Use anonymized/synthetic test data only.
RULE 3: CODE REVIEW IS MANDATORY All Claude-assisted code must go through peer review before merging. You cannot review your own code. Skipping code review results in automatic PR rejection.
RULE 4: UNDERSTAND CODE BEFORE COMMITTING You must thoroughly read and understand all code Claude generates before it touches git. If you don't understand it, don't commit it.
RULE 5: STAGING FIRST, ALWAYS All code deploys to staging environment first. Production deployment requires explicit approval from tech lead after staging validation.
RULE 6: AGENTS ARE READ-ONLY BY DEFAULT Custom subagents must have restricted permissions. Full write access requires explicit justification and approval. Agent permissions reviewed quarterly.
RULE 7: NO HARDCODED VALUES IN CODE Configuration, secrets, and environment-specific values come from environment variables, .env files, or secret vaults. Never hardcode them.

3. Security Hardening Strategy

Security strategy uses defense-in-depth layering against real attack vectors.

Threat 1: Secret Leakage to Claude

Impact: Attacker obtains API key → uses key to access production → $50k-$500k fraudulent charges, data breach, legal liability.

Defense Layers:

Layer 1 - Developer Training: All developers trained on secret recognition. Never paste .env files, credentials, or tokens into Claude. Use placeholders in all prompts.
Layer 2 - Pre-Commit Hooks: Local git hooks scan commits for secret patterns: API_KEY=, password=, sk_live_, AWS key patterns. Blocks commits with detected secrets.
Layer 3 - GitHub Secret Scanning: GitHub automatically scans all push events for known credential patterns. Alerts security team immediately if secrets detected.
Layer 4 - Monthly Audit: Git history scanned monthly with specialized tools (TruffleHog). Any leaked credentials found trigger immediate rotation + investigation.
Layer 5 - Secret Rotation: All production secrets rotated quarterly automatically. If leak suspected, rotated within 1 hour + API audit trail reviewed.
Layer 6 - Vault Storage: Production secrets stored ONLY in AWS Secrets Manager or HashiCorp Vault. Never in code, config files, or filesystem env vars.

Threat 2: Production Data Exposure

Impact: Developer shares production database dump with Claude → GDPR/CCPA violation → €20M fines, customer lawsuit, reputation damage.

Defense Layers:

Layer 1 - Access Control: Only specific roles have production database access. Junior developers have zero production database access.
Layer 2 - Anonymization: If working with data, use anonymized/synthetic datasets. Queries return no real customer identifiers.
Layer 3 - Audit Logging: All production database access logged with timestamp, user, query, and result count. Suspicious access patterns trigger alerts.
Layer 4 - Manual Review: Any data-related Claude prompt reviewed by team lead before execution. No automated scanning of prod data.

Threat 3: Code Vulnerabilities

Impact: Claude generates code with security flaw → SQL injection, XSS, auth bypass → production compromise.

Defense Layers:

Layer 1 - Code Review: All code reviewed by another developer for security issues: input validation, error handling, auth checks, OWASP compliance.
Layer 2 - Static Analysis: SAST tools (Snyk, SonarQube) scan all code commits. Known vulnerabilities block merges.
Layer 3 - CI/CD Testing: Tests must pass. Security tests included: XSS, SQL injection, auth tests.
Layer 4 - Staging Validation: Code validated in staging environment before production. Security team can perform additional testing.

Threat 4: Rogue or Misconfigured Subagents

Impact: Subagent with full write access makes unauthorized changes → production data modified/deleted.

Defense Layers:

Layer 1 - Restricted Permissions: Subagents created with minimal tool access by default. Read-only for validators/reviewers. Write access requires justification.
Layer 2 - Approval Process: New subagents with write access require tech lead approval before creation.
Layer 3 - Test-Only Data: Agents work with test databases/test environments only. Production connection strings stored in vault, inaccessible to agents.
Layer 4 - Audit Logging: All agent actions logged. Suspicious patterns (bulk deletes, config changes, data exports) trigger alerts.
Layer 5 - Regular Audit: Subagents audited quarterly. Unused agents deleted. Permissions reviewed and reduced to minimum necessary.

4. Development Workflow

Standard workflow when using Claude Code for development.

Step-by-Step Workflow (Full Subagent Pipeline: Arch → Write → Verify)

Best Practice: Use specialized subagents at each stage. Tech-Architect designs, Tech-Writer documents, Senior-Engineer implements, Verification subagents confirm. This is the highest-quality development pattern.

Define Business Requirements Describe what feature needs to do from business perspective. Who uses it? What problems does it solve? Example: "Users need to reset forgotten passwords. Current process is broken—too many support requests."
2

Output: Detailed technical specification that senior-engineer will implement.
3

4

Senior-engineer has access to: project context (CLAUDE.md), codebase standards, architecture patterns. Implements exactly to spec—architecture already designed, requirements already documented, implementation is straightforward. Includes: tests, error handling, documentation, security built-in.
5
Code Quality Verification Use /invoke code-reviewer to verify: naming conventions, code structure, performance, adherence to project patterns. Should find minimal issues (senior-engineer implemented to spec, code is already good).
6
Security & Coverage Verification Use /invoke security-auditor to confirm: no SQL injection, auth/authz correct, no secrets in code, data protection. Use /invoke test-coordinator to verify: all logic paths covered by tests, spec's acceptance criteria covered. Verification only—should pass cleanly.
7
Local Testing & Validation Run all tests locally (unit, integration, manual). Verify spec's acceptance criteria fully met. Confirm all edge cases handled. If issues found, request senior-engineer to fix (not manual patches).
8
Create Feature Branch & Commit git checkout -b feature/description. Commit with meaningful message. One feature per commit when possible. Code is production-ready—designed by architect, specified by writer, implemented by senior engineer, verified by reviewers.
9
Create Pull Request Push to remote. Create PR with: tech-architect's design decisions, tech-writer's acceptance criteria, senior-engineer's implementation, test coverage achieved, security verification results.
10
Manual Architectural Review (Final Gate) Tech lead reviews: does implementation match tech-architect's design? Are architectural decisions sound? Does it align with project architecture? Not reviewing: coding standards (code-reviewer verified), security basics (security-auditor verified), test coverage (test-coordinator verified). This is final approval before UAT.
11
Address Architectural Feedback Review tech lead comments. Make requested design/architecture changes. Re-run senior-engineer if major changes needed. Push new commits.

13
Approval & Merge Tech lead approves. Senior developer merges to main/develop branch.
14

15
Production Deployment Tech lead approves after staging validation. Code deployed to production. Monitoring activated.

Five-Phase Subagent Pipeline (Arch → Spec → Code → Verify → Approve)

Why this approach? Each specialist subagent does one job perfectly. Architecture is sound, requirements are clear, code is right, quality is verified, decisions are reviewed. Result: Highest-quality code with zero rework.

Phase 1: Tech-Architect Designs Solution (Step 2)
  • Tool: /invoke tech-architect
  • Input: Business requirements, project context (CLAUDE.md), existing architecture
  • Output: Technical specification with:
    • System design and architecture decisions
    • Integration points and data flow
    • Database schema or API changes needed
    • Security architecture and constraints
    • Performance requirements and scale
    • Edge cases and error scenarios
  • Why: Prevents bad architecture decisions that are expensive to fix later
Phase 2: Tech-Writer Documents Specification (Step 3)
  • Tool: /invoke tech-writer
  • Input: Tech-architect's design, business requirements
  • Output: Clear specification with:
    • Acceptance criteria (testable requirements)
    • Test scenarios (happy path + error cases)
    • Data validation rules
    • Error handling requirements
    • Edge cases to handle
    • Success metrics
  • Why: Senior-engineer implements against clear contract, not ambiguous requirements
Phase 3: Senior-Engineer Implements (Step 4)
  • Tool: /invoke senior-engineer
  • Input: Tech-architect's design + Tech-writer's specification
  • Output: Production-ready code with:
    • Implementation following design exactly
    • Comprehensive unit and integration tests
    • All acceptance criteria from spec implemented
    • Security checks and input validation
    • Error handling with meaningful messages
    • Documentation for complex logic
  • Why: Clear specification → fewer implementation mistakes → fewer revisions
Phase 4: Verification (Steps 5-6)
  • Tools: code-reviewer, security-auditor, test-coordinator
  • Purpose: Confirm:
    • Code quality and project standards met
    • Security requirements verified
    • Test coverage matches acceptance criteria
  • Expected: All verifications pass cleanly. Issues are rare (code already good).
Phase 5: Architectural Review (Step 10, Final Gate)
  • Reviewer: Tech lead / senior architect (manual human review)
  • Purpose: Verify implementation matches tech-architect's design decisions
  • NOT Reviewing: Coding standards (code-reviewer verified), security basics (security-auditor verified), test coverage (test-coordinator verified)
  • Focus: Does implementation align with design? Are architectural patterns consistent? Is business logic correct?

Why This Five-Phase Pipeline?

Each specialist does one thing perfectly:
  • Tech-Architect: Ensures technical decisions are sound before code is written
  • Tech-Writer: Ensures requirements are crystal clear before implementation starts
  • Senior-Engineer: Implements against clear spec, high quality output
  • Code/Security/Test Reviewers: Verify standards met (should find minimal issues)
  • Tech Lead: Reviews only architecture (not coding style or security basics)
Quality multiplier effect:
  • Bad designs caught in Phase 1 (cheap to fix)
  • Ambiguous requirements caught in Phase 2 (cheap to fix)
  • Implementation mistakes rare in Phase 3 (clear spec)
  • Verification finds minimal issues in Phase 4 (code already good)
  • Architectural review smooth in Phase 5 (design already approved)

Five Approaches Compared

LEVEL 0 (Weakest):
Write Code → Commit → PR → Manual Review → UAT finds bugs
→ Result: Bugs reach staging/UAT

LEVEL 1:
Write Code → Verify → Commit → PR → Manual Review → UAT
→ Result: Better, but design flaws still in PR

LEVEL 2:
Architect → Write Code → Verify → Commit → PR → Manual Review → UAT
→ Result: Good, but requirements still unclear

LEVEL 3:
Architect → Write Spec → Write Code → Verify → Commit → PR → Manual Review → UAT
→ Result: Very good, one specialist per stage

LEVEL 4 (Best - Full Pipeline):
Arch Designs → Writer Specs → Engineer Codes → Verify Quality → Manual Review
→ Result: Highest quality, zero rework, each specialist perfect at their job

Key Points - Five-Phase Subagent Pipeline

  • Tech-Architect designs FIRST (Step 2): Ensures technical approach is sound before any code is written. Catches design flaws early (cheap to fix)
  • Tech-Writer specifies SECOND (Step 3): Ensures requirements are crystal clear. Acceptance criteria become the implementation contract.
  • Senior-Engineer implements THIRD (Step 4): Implements against clear spec and proven design. Code is high-quality from the start.
  • Verification subagents confirm FOURTH (Steps 5-6): Verify quality/security/coverage. Should find minimal issues (code already good).
  • Manual architectural review is FIFTH (Step 10): Tech lead confirms implementation matches design. NOT checking coding style or security basics.
  • Do NOT skip any phase: Each phase prevents different categories of bugs
  • Do NOT write code without architect approval: Bad design is expensive to fix later
  • Do NOT implement without tech-writer spec: Ambiguous requirements cause rework
  • Do NOT skip verification steps: Confirm quality/security even with good code
  • Do NOT have single Claude write entire feature: Use specialized subagents (architect, writer, engineer)

  • Verification is fast - Code should be good from architect guidance and writer spec, verification just confirms

What Is MCP?

Model Context Protocol is a standard that lets Claude Code connect to external tools, services, and APIs. It enables Claude to interact directly with GitHub, databases, APIs, and other systems.

Why is MCP Required?

How MCP Works (Simple Explanation)

Context Mode MCP - Why It's Required

Context Mode is MANDATORY for all Claude Code users. It solves a critical problem in AI-assisted development: context window exhaustion.

The Problem Without Context Mode

When you work with Claude for extended periods (2+ hours), context gets consumed rapidly:

  • Large tool outputs dump raw data into your context (Playwright screenshots: 56 KB, GitHub issues: 59 KB, logs: 45 KB)
  • After 30-40 minutes, 40% of your available context is consumed
  • Claude loses track of what you're working on and previous decisions
  • You have to start new conversations, losing all previous context
  • Productivity drops significantly on long-running tasks

What Context Mode Does

  • Reduces context usage by 98%: Converts 315 KB outputs to 5.4 KB summaries using sandbox processing
  • Maintains session continuity: Tracks all file edits, git operations, errors in local SQLite database
  • Knowledge base search: Full-text search across documentation with fuzzy matching
  • Local-only processing: No data leaves your machine (100% privacy)

Context Mode operates transparently - you don't need to invoke it manually. It automatically:

  • Maintains session state for long conversations
  • Provides ctx_search command for knowledge base queries

  • Security Auditor: Read-only security scanning
  • Performance Optimizer: Analyze and suggest optimizations (read-only)
  • Documentation Writer: Generate/update documentation

Model Selection Guide: Which Model to Use

Choosing the right model for each subagent is critical for cost, speed, and quality.

Model Overview

Model Speed Cost Reasoning Best For
claude-haiku-4-5 Very Fast $ Small, fast, lightweight reasoning Quick validation, basic analysis, repetitive tasks
claude-sonnet-4-5 Medium $$ Balanced speed & capability General implementation, testing, standard reviews

Subagent-by-Subagent Model Selection

Detailed recommendations for each type of subagent.

1. Code Reviewer Subagent

Recommended Model: claude-haiku-4-5

Why Haiku?

  • Code review is predictable: check style, security, error handling
  • Fast response needed (developer waiting for feedback)
  • Read-only access - no risky decisions
  • Pattern matching - Haiku excels at finding known issues
  • Cost: Reviewing code is frequent, Haiku saves money

When to use Opus instead: Complex architectural reviews, performance analysis, or when pattern is unusual

2. Security Auditor Subagent

Recommended Model: claude-opus-4-6

Why Opus?

  • Security requires deep reasoning: attackers are creative
  • False negatives are expensive (missed vulnerabilities)
  • Complex threat modeling needed
  • OWASP, CWE, and nuanced security patterns
  • Cost justified: security breaches cost millions

When to use Haiku: Quick syntax checks only, not comprehensive audits

3. Performance Optimizer Subagent

Recommended Model: claude-opus-4-6

Why Opus?

  • Optimization requires understanding system design
  • N+1 queries, caching strategies, algorithm analysis
  • Complex tradeoffs: speed vs memory vs maintainability
  • Haiku might miss subtle performance issues

When to use Haiku: Simple pattern checks (e.g., "find all loops in file")

4. Documentation Writer Subagent

Recommended Model: claude-haiku-4-5

Why Haiku?

  • Documentation is mostly copy + structure
  • Haiku is fast - rapid doc generation
  • Code comments, README, API docs - predictable format
  • Can always improve with human review
  • Cost: Documentation is frequent, volume is large

When to use Opus: Complex architecture documentation, technical deep-dives

5. Test Generator Subagent

Recommended Model: claude-haiku-4-5

Why Haiku?

  • Test generation is patterned (setup → act → assert)
  • Haiku knows Jest, Pytest, PHPUnit patterns well
  • Fast generation - developers need tests quickly
  • Human reviews generated tests anyway
  • Cost: Test generation is frequent and high-volume

When to use Opus: Complex integration test scenarios, edge case coverage strategies

6. Bug Investigator / Debugger Subagent

Recommended Model: claude-opus-4-6

Why Opus?

  • Debugging requires reasoning through code logic
  • Multiple possible root causes - need smart hypothesis
  • Error messages often misleading - need deep analysis
  • Haiku might blame the wrong component
  • Cost justified: Developer time waiting for answer is expensive

7. Database Query Optimizer Subagent

Recommended Model: claude-opus-4-6

Why Opus?

  • SQL optimization requires understanding query plans
  • Index strategies, join optimization, indexing decisions
  • Wrong optimization can slow system down
  • Haiku might suggest obviously inefficient queries

8. Architecture/Design Reviewer Subagent

Recommended Model: claude-opus-4-6

Why Opus?

  • Architecture decisions affect entire system
  • Must evaluate SOLID, DDD, design patterns
  • Scaling implications, maintainability tradeoffs
  • Cost: Bad architecture costs months to refactor

9. Commit Message Generator Subagent

Recommended Model: claude-haiku-4-5

Why Haiku?

  • Commit messages are short, structured format
  • Pattern: "type: description" is simple
  • Fast generation - for frequent commits
  • Can always edit if not perfect

10. API Contract Validator Subagent

Recommended Model: claude-sonnet-4-5

Why Sonnet?

  • API contracts are formal but nuanced
  • Request/response schema validation
  • Error handling expectations
  • Haiku might miss edge cases, Opus is overkill
  • Sonnet is the sweet spot for cost/capability

Decision Matrix: Quick Reference

Task Type Model Cost Why
Quick pattern check (read-only) Haiku Low Fast, accurate for simple patterns
Code generation, tests, docs Haiku Low Predictable format, human reviews output
General feature implementation inherit (Sonnet) Medium Balanced speed/capability, most tasks

Cost Optimization Strategy

  • Default to Haiku: Use for read-only validation, pattern matching, generation
  • Use Sonnet for: Most implementation, balanced tasks
  • Reserve Opus for: Security, performance, architecture, complex reasoning
  • Use "inherit": Let CLAUDE.md project default decide (recommended for 80% of agents)
  • Monitor costs: Opus costs 10x more than Haiku - use judiciously

Advanced Pattern: Multi-Agent Orchestration for Complex Tasks

For large projects, deploy multiple specialized agents in parallel to handle different aspects of the problem.

When to Use Multi-Agent Orchestration

  • Complex features requiring analysis from multiple domains
  • Architecture decisions needing different expert perspectives
  • Large-scale refactoring or system redesigns
  • Requirements that need validation from multiple angles

Agent Roles in Complex Tasks

Role Responsibility Model

Architect Design HOW Opus

Multi-Agent Workflow

1. Orchestrator receives task
2. Split into domain-specific sub-tasks
3. Deploy agents in parallel (Requirements, Architecture, Security, Performance, QA)
4. Each agent analyzes independently
5. Orchestrator synthesizes findings
6. Create unified output/specification

Key Rules for Multi-Agent Systems

  • Agent Boundaries: Each agent outputs text only (no direct file writes)
  • Orchestrator Controls Writes: Only orchestrator/human modifies files

  • Comprehension Gates: Verify agents understand context before proceeding
  • Audit Trail: Document each agent's analysis for traceability
  • Drift Detection: Monitor gaps between specification and actual implementation

Preventing "Skill Atrophy"

When complex contexts build up over time, AI models can lose grasp of the full specification. Prevent this with:

  • Regular Comprehension Checks: Agents re-validate understanding of specs
  • Documentation Sync: Keep specs synchronized with code automatically
  • Periodic Re-briefing: For long projects, update agent context regularly
  • Structured Formats: Use machine-readable specs (not free-form) for clarity

Why Skills Matter for Subagents

2. Extract ONLY relevant method (~60 lines max)

4. Refactor bounded scope

6. Re-integrate into codebase

4. Regression: Ensure existing features still work

## Anti-Pattern Detection
- Detect circular tests (tests that pass because they mirror code)
- Verify tests check constraints, not just "feature exists"
- Ensure edge cases covered (empty states, timeouts, failures)

5. Tech-Writer Agent → specification-clarity-skill

Ensures intended-docs.md specs are testable and unambiguous. Prevents AI hallucination by enforcing precision.

## Specification Clarity Checklist
- [ ] All requirements testable (not subjective language)
- [ ] Acceptance criteria specific (numbers, not "fast")
- [ ] Edge cases defined (expired tokens, errors)
- [ ] Performance has numbers (< 500ms, not "quickly")
- [ ] Security explicit (not implied)
- [ ] No vague language: "happy", "intuitive", "optimized"

8.2 Fast-Track Workflow (Low-Risk Changes Only)

The 5-phase pipeline creates friction for small, low-risk changes. Use the Fast-Track workflow for changes that:

  • Touch a single file (no cross-module dependencies)
  • Are clearly scoped (e.g., "fix typo in error message" OR "change CSS padding")
  • Don't change database schema, API contracts, or security
  • Have no third-party integrations involved
  • Are reversible (no data migration, no breaking changes)

Fast-Track Process (Tech-Lead Only)

1. Define change in one sentence
   "Change button color from red to blue in login form"

2. Create branch and implement
   $ git checkout -b feature/button-color

3. Write minimal test
   // Verify button renders with new color
   test('login button is blue', () => {
     expect(button.getComputedStyle('color')).toBe('blue');
   });

4. Run tests locally
   $ composer test
   $ npm test

5. Self-review: Check diff
   $ git diff main

6. Tech-Lead reviews (ONE human review, not 5-phase)
   [ ] Change matches description?
   [ ] Tests pass locally?
   [ ] No secrets/credentials exposed?
   [ ] Doesn't break other features?

7. If approved: Merge and deploy
   $ git push origin feature/button-color
   # Tech-Lead merges and deploys directly

Total time: 30 minutes vs 3-5 hours for full pipeline.

Fast-Track Rejection Triggers

If ANY of these are true, STOP and use full 5-phase pipeline:

  • Tech-Lead is uncertain about the change
  • Change touches business logic (not just UI/style)
  • Affects API contracts or data models
  • Involves third-party service (AWS, email, SMS)
  • Change is part of a larger feature (use full pipeline instead)
  • Tests fail or are missing

Fast-Track is NOT an exception process - it's a parallel track for genuinely low-risk changes.

8.4 CI/CD Failure Workflows & Debugging Deep-Dives

Problem: Code passes local tests but fails in CI/CD (Jenkins, CircleCI) due to infrastructure differences, environment variables, or integration issues.

Pattern: Differential Debugging (Isolate Root Cause)

Scenario:
  - Local: All tests PASS (composer test ✓)
  - CI: Tests FAIL (CircleCI ✗)
  - Staging: Feature works ✓

Root cause unknown. How to find it?

Solution: Differential Debugging

Step 1: Identify what's different between environments
Local:
  - PHP 8.2
  - SQLite in-memory database
  - No AWS S3 (file system mocked)
  - No email service (mail faked)

CI:
  - PHP 8.1 (different!)
  - PostgreSQL (different!)
  - AWS S3 required
  - AWS SES required
  - Different environment variables

Step 2: Reproduce CI environment locally
  - Install PHP 8.1
  - Set up PostgreSQL locally
  - Configure AWS credentials
  - Copy .env.ci to .env.local
  - Run tests: composer test
  - If still fails locally, you found the issue

Step 3: Invoke Debug Agent
/clear (start fresh context)
/invoke code-reviewer
"Debug this test failure:

Local tests pass:
  $ composer test
  ✓ All 24 tests pass

CI tests fail:
  $ circleCI logs show:
  ✗ PasswordResetTest: timeout after 30s
  ✗ S3FileUpload: 403 Forbidden
  ✗ SESEmailSend: Auth failed

Environments:
  Local: PHP 8.2 + SQLite + Mocked AWS
  CI: PHP 8.1 + PostgreSQL + Real AWS

Diff of Main vs Feature:
  [paste git diff output]

Find the root cause and suggest fix."

Step 4: Apply fix and test
  - Make the fix locally
  - Run tests locally: composer test
  - Push to feature branch
  - Watch CI logs: did it fix the issue?

Common CI/CD Failure Patterns

1. Timeout Issues
   Symptom: "Test timeout after 30s"
   Usual cause: Database queries too slow, or query hangs
   Fix: Profile database queries, add indexes, mock slow services

2. Permission Errors (403, 401)
   Symptom: "AWS S3 403 Forbidden"
   Usual cause: AWS credentials missing or wrong in CI environment
   Fix: Verify IAM role, AWS_ACCESS_KEY, AWS_SECRET in CI config

3. Database Migration Failures
   Symptom: "MIGRATION FAILED: Column already exists"
   Usual cause: Multiple CI runs trying to migrate same database
   Fix: Use database locking or check if migration already applied

4. Port/Service Conflicts
   Symptom: "Cannot connect to Redis on localhost:6379"
   Usual cause: Service not started in CI, or port in use
   Fix: Ensure Redis/Postgres/etc. started before tests in CI

5. Environment Variable Mismatches
   Symptom: Tests pass locally, fail in CI
   Usual cause: .env.local has values, .env.ci missing them
   Fix: Use same .env file for local and CI (no local overrides)

6. Race Conditions
   Symptom: "Flaky tests - pass sometimes, fail others"
   Usual cause: Tests not isolated, or timing dependent
   Fix: Add explicit waits, mock timers, ensure test isolation

Mandatory CI/CD Checklist Before Merging

[ ] Code passes local tests (composer test)
[ ] Code passes CI tests (CircleCI/Jenkins shows green)
[ ] No skipped tests (@skip annotations removed)
[ ] Test coverage >= 80% for new code
[ ] No flaky tests (run CI 3 times, all pass)
[ ] All environment-specific configs documented
[ ] Rollback procedure tested
[ ] No hard-coded credentials in code or CI logs
[ ] Performance hasn't degraded (benchmark main vs feature)

If CI fails:
  1. Do NOT merge
  2. Investigate using Differential Debugging
  3. Fix root cause
  4. Re-run CI until green
  5. Then merge

Risk: Claude reads malicious code/comments that override CLAUDE.md instructions or trick the model into executing unsafe commands.

10. Deployment Procedures

Deployment Pipeline

All code goes through this pipeline before production:

1
Developer Merge Code approved, CI/CD passes, senior developer merges to main branch
2
Automated CI/CD Tests run automatically, linting checked, code scanned for vulnerabilities, coverage verified
3
Staging Deploy Automatically deployed to staging environment. Staging uses same database schema, same external services (test versions).
4
Staging Validation QA team and developers test in staging. Smoke tests run. Security team may conduct additional testing.
5
Production Approval Tech lead reviews staging test results. Approves for production deployment.
6
Production Deploy Code deployed to production. Blue-green deployment used (new version runs alongside old, traffic switched when ready).
7
Monitoring Alerts and monitoring activated. Team watches for errors, performance issues, crashes.

Zero-Risk Deployment Checklist

  • [ ] All tests passing (unit, integration, security)
  • [ ] Code reviewed and approved by second developer
  • [ ] Static analysis (Snyk, SonarQube) found no critical issues
  • [ ] Staging deployment successful
  • [ ] Staging testing completed without issues
  • [ ] Tech lead approval obtained
  • [ ] Monitoring and alerts configured
  • [ ] Rollback procedure documented and tested
  • [ ] Team available for monitoring after deployment

Forbidden Deployment Practices

  • Deploying code you haven't personally reviewed
  • Deploying without tests passing
  • Skipping staging environment
  • Deploying Friday evening/weekend without team available
  • Deploying code with known vulnerabilities
  • Force-pushing to main/production branches
  • Modifying production database manually from deployments

Rollback Procedure

If production deployment causes issues:

  1. Tech lead notified immediately
  2. Gather error logs and monitoring data
  3. Assess scope of issue (user-facing? data-affecting? security?)
  4. If critical: Rollback to previous version (revert deployment)
  5. Create incident and post-mortem
  6. Fix issue and re-deploy through normal process

11. Incident Response

Incident Types and Response Times

Incident: Suspected Secret Leakage

Response Time: 1 Hour

If you suspect a secret has been shared with Claude, committed to git, or exposed anywhere:

  1. Notify security team immediately (do not delay)
  2. Rotate the secret/key immediately (AWS, API provider, database)
  3. Search git history for the secret
  4. Determine if secret was used before rotation
  5. Review API/database audit logs for suspicious activity
  6. If unauthorized access found, escalate to incident response team

Incident: Production Vulnerability Discovered

Response Time: 4 Hours

If code with security vulnerability (SQL injection, XSS, auth bypass) deployed to production:

  1. Create critical bug report
  2. Assess user impact (are users affected?)
  3. Determine if vulnerability exploitable (is it actually a problem in practice?)
  4. Fix vulnerability (write patch code)
  5. Code review and test patch urgently
  6. Deploy patch to staging first, then production
  7. If user data affected, notify users and tech lead
  8. Conduct post-mortem to prevent recurrence

Incident: Data Exposure (Production Data Shared)

Response Time: 30 Minutes

If production data (customer records, PII, transactions) exposed to Claude, GitHub, or external service:

  1. Notify tech lead and security team immediately
  2. If in Claude: notify Anthropic support for data deletion
  3. If in git: determine if visible publicly or private
  4. Assess scope: how much data? how sensitive? which customers?
  5. If GDPR/CCPA applies: prepare user notification (legal will advise)
  6. Remove data from systems where exposed
  7. Conduct investigation and post-mortem

Post-Incident Actions

After any security incident:

  • Conduct post-mortem meeting within 1 week
  • Document root cause: how did this happen?
  • Identify control failure: which layer of defense failed?
  • Implement fix: how do we prevent this?
  • Update training: what should team know?
  • Monitor: watch for similar issues in future

12. FAQ & Troubleshooting

Frequently Asked Questions

Q: Can I share my .env file with Claude to get help debugging?

A: No. Use a .env.example file with placeholder values instead. Share the placeholder file, not the real .env with actual values.

Q: What if I accidentally paste a secret into a Claude prompt?

A: Rotate the secret immediately (AWS, API provider, etc.). Then notify security team. Do not delay.

Q: Can I use production database for testing Claude code?

A: No. Always use test/staging database. If you need to test with production-like data, use anonymized/sanitized data only.

Q: My subagent needs write access to deploy code. Is that allowed?

A: Subagents should not deploy code directly. Instead, subagent creates code that humans review and deploy manually through CI/CD pipeline.

Q: Can I approve my own code?

A: No. Code must be reviewed by another developer. This is non-negotiable.

Q: What if my code review takes longer than usual?

A: That's fine. Thorough review is more important than speed. Request help from another reviewer if needed, but do not skip review.

Q: Can I merge to main branch directly?

A: No. All code goes through PR with review process. Direct merges are only for tech leads in emergency situations.

Q: What if Claude suggests hardcoded configuration values?

A: Modify code to use environment variables or config files instead. Store actual values in .env or AWS Secrets Manager.

Q: How do I know if a subagent has too much access?

A: If subagent has write access, ask: Is it necessary? Can it be read-only instead? If write access needed, does it have production database access? (Should be test database only)

Q: What if I find a bug in Claude-generated code after deployment?

A: Create critical bug report immediately. Fix code urgently. Deploy patch through normal process (PR → review → staging → production).

Common Issues

Issue: Pre-commit hook blocks my commit with "secret detected"

Solution: Check the flagged lines. Remove any real secrets. If it's a false positive (e.g., test string that looks like secret), whitelist the pattern with tech lead.

Issue: GitHub secret scanning found credentials in old commit

Solution: Rotate the credential immediately. Use BFG Repo-Cleaner to remove from git history if needed. Notify security team.

Issue: Code review is taking too long

Solution: Request another reviewer's help. Break PR into smaller changes if it's too large. Do not merge without review.

Issue: My subagent keeps failing on certain tasks

Solution: Review subagent instructions and tool permissions. Test with simpler tasks first. Check if subagent has necessary tools (Read, Bash, etc.).

Issue: Test failures in CI/CD pipeline

Solution: Run tests locally with same test framework. Debug locally. Push fix to branch. CI/CD runs again automatically.
