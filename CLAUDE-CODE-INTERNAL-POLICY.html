1. Introduction & Purpose

This document establishes mandatory policies for using Claude Code (and similar AI coding assistants) across the development organization.

1.1 Purpose of Using AI in Development

Claude Code is approved for use to:

  • Accelerate development - Faster feature implementation with proper review
  • Improve code quality - Better patterns, architecture, and practices
  • Reduce manual work - Testing, documentation, refactoring
  • Knowledge transfer - Explaining patterns and best practices
  • Problem solving - Debugging and optimization

Claude Code is NOT a replacement for human judgment, security review, or testing.

1.2 Risks & Responsibilities

Using AI in development carries specific risks that YOU must mitigate:

  • Security Leaks: Accidentally sharing secrets, API keys, credentials
  • Code Quality: AI-generated code may contain bugs, performance issues, security vulnerabilities
  • Blind Trust: Accepting code without understanding or testing it
  • Production Impact: Bad code reaching production and causing incidents
  • Intellectual Property: Training data privacy and model limitations
  • Compliance: Regulatory violations (PCI-DSS, HIPAA, GDPR)
⚠️ CRITICAL: You are legally and professionally responsible for code you commit, regardless of whether Claude generated it.

AI-assisted development does NOT reduce your code review obligations. If Claude generates vulnerable code and you commit it without testing, YOU are accountable.

2. Security Policy (STRICT RULES)

This is the most critical section. Violations will result in immediate access revocation and disciplinary action.

2.1 Strictly Prohibited: Secrets & Credentials

🛑 NEVER share with Claude Code:
  • API keys or tokens (Auth0, Stripe, AWS, etc.)
  • Database connection strings with passwords
  • OAuth client secrets
  • JWT signing keys or refresh tokens
  • SSH keys or private keys
  • AWS credentials or access keys
  • Certificate private keys
  • Encryption keys or salt values
  • Third-party service passwords
  • Slack/GitHub tokens or personal access tokens

Examples of VIOLATIONS:

❌ BAD - VIOLATION:

"Claude, here's my .env file. Why is my API not working?"

STRIPE_API_KEY=sk_live_4eC39HqLyjWDarhtT657pxB
DATABASE_URL=postgresql://admin:P@ssw0rd123@prod-db.rds.amazonaws.com:5432/myapp
AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
❌ BAD - VIOLATION:

"Can you review my OAuth implementation? Here's the client secret:"

client_id: auth0_abc123xyz
client_secret: xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx_PRODUCTION
✅ CORRECT - SAFE:

"Why is my API not working? Here's my code (with secrets removed):"

// .env.example (SAFE - no actual values)
STRIPE_API_KEY=sk_test_XXXXXXXXXXXXX
DATABASE_URL=postgresql://user:XXXXX@localhost:5432/myapp
AWS_SECRET_ACCESS_KEY=XXXXXXXXXXXXXXXXXXXXXXXXXXXXXX

// Code reference:
const stripe = require('stripe')(process.env.STRIPE_API_KEY);
✅ CORRECT - SAFE:

"I'm having issues with OAuth. Here's the flow (without secrets):"

// Implementation pattern (no actual secrets)
const auth0 = new Auth0Client({
  domain: 'YOUR_DOMAIN.auth0.com',
  client_id: 'YOUR_CLIENT_ID',
  // Do NOT share client_secret with Claude
});

// Ask Claude for help with flow, not secrets

What To Do Instead:

  • Remove secrets before sharing: Use .example files or placeholder values
  • Ask about the pattern, not the secret: "How do I securely store API keys?" (general question)
  • Ask locally: Describe the problem without sharing actual credentials
  • Use dummy values: "Here's my setup using dummy values: sk_test_xxxxx"

2.2 Strictly Prohibited: Production Data

🛑 NEVER share with Claude Code:
  • Real customer data, names, emails, phone numbers
  • Real payment card information or financial data
  • Database backups or production database dumps
  • User PII (Personally Identifiable Information)
  • Health information, medical records, or sensitive personal data
  • Real error logs with sensitive information
  • Production credentials, IPs, or infrastructure details
  • Real transaction IDs or user IDs that identify individuals
❌ BAD - VIOLATION (Privacy/Compliance):

"Debug this user query. Here's the actual data:"

SELECT * FROM users WHERE id = 45328;
Results:
- Name: John Smith
- Email: john.smith@gmail.com
- Phone: +1-555-0123
- SSN: 123-45-6789
- Credit Card: 4532-1234-5678-9010
✅ CORRECT - SAFE:

"How do I optimize this user query?"

SELECT * FROM users WHERE id = ?;
Results: 1 row with fields: id, name, email, created_at

Question: The query takes 2 seconds for 100k rows.
How can I optimize it with proper indexing?

Compliance Regulations:

  • GDPR: Cannot share EU customer data with external AI (includes names, emails)
  • HIPAA: Cannot share health information with any external service
  • PCI-DSS: Cannot share payment card information
  • SOC 2: May restrict sharing of internal infrastructure details

2.3 Handling Secrets in Code

Rule: NEVER commit secrets to ANY branch

Configuration Management:
  • Local Development: Use .env files (in .gitignore)
  • Staging/Production: Use AWS Secrets Manager or environment variables
  • Version Control: Only commit .env.example with placeholders
  • Code Review: PR reviewers must verify no secrets are present
✅ CORRECT - Safe Setup: 
// .env (never committed - in .gitignore)
DATABASE_URL=postgresql://user:pass@localhost:5432/myapp
API_KEY=sk_live_xxxxxx

// .env.example (safe - committed to repo)
DATABASE_URL=postgresql://user:PASSWORD@localhost:5432/myapp
API_KEY=sk_live_xxxxx

// Code loads from environment
const dbUrl = process.env.DATABASE_URL; // Loaded from .env

If You Accidentally Share a Secret:

  1. Stop immediately - Do not continue the conversation
  2. Notify Security Team - Email: security@company.com within 1 hour
  3. Rotate the secret - Change API key, password, token immediately
  4. Check git history - Verify it's not in any git commits
  5. Update CLAUDE.md - Document what happened for transparency

2.4 Real-World Security Risk Scenarios

Scenario 1: The "Just One Secret" Mistake

What Happened:

Junior developer shares production Stripe key with Claude to "debug payment issue". Claude generates code using that key. Key gets stored in training data. Attacker finds it and steals $50k in fraudulent charges.

Consequences: Developer loses access. Company pays refunds. PCI audit fails.

Scenario 2: The .env File Paste

What Happened:

Developer asks Claude: "Why is database connection failing?" and pastes entire .env file. AWS secret key, database password, encryption keys all exposed. Attacker gains database access.

Consequences: Data breach. Customer notification required. Legal liability.

Scenario 3: The "Test Data" Leak

What Happened:

Developer shares "test database dump" with Claude to "optimize query". It's actually production data with 10k customer emails and payment info. GDPR violation detected.

Consequences: €20M GDPR fine. Company reputation damage.

Scenario 4: The GitHub Token Mistake

What Happened:

Developer shares error log with Claude. Error log contains GITHUB_TOKEN from CI/CD pipeline. Attacker uses token to push malicious code to repositories.

Consequences: Supply chain compromise. Malware in production.

How to Prevent These Scenarios:

  • ✓ Create .env files with placeholder values BEFORE using Claude
  • ✓ Use a script to redact sensitive data from logs/errors
  • ✓ Keep production credentials in secure vaults (AWS Secrets Manager, HashiCorp Vault)
  • ✓ Use separate credentials for development vs production
  • ✓ Enable secret scanning in GitHub to catch leaked credentials
  • ✓ Audit Claude conversations for accidental leaks (monthly)

4. Development Workflow

The mandatory process for using Claude Code in your development lifecycle.

4.1 The Process: Claude → Local → Branch → PR → Review → CI → Deploy

Step 1: Use Claude Code (Local Environment Only)

✓ Always use Claude Code locally, never in production environment

✓ No sensitive credentials in your prompts

✓ Test all code locally before committing

Step 2: Create Feature Branch

✓ Branch name: feature/descriptive-name or bugfix/issue-number

✓ Never commit to main or develop directly

✓ Branch should be short-lived (max 1 week)

Step 3: Verify Code Locally

✓ Run all linting: npm run lint

✓ Run all tests: npm test

✓ Build successfully: npm run build

✓ Test manually (feature works correctly)

✓ Check for security issues (see Section 6)

Step 4: Create Commit with Claude

✓ Use /commit skill to generate commit message

✓ Or write meaningful commit message manually

✓ Reference issue number: "Closes #123" or "Fixes #456"

Step 5: Push to Remote Branch

✓ git push origin feature/your-branch

✓ Never force push to shared branches

✓ Verify GitHub shows your branch

Step 6: Create Pull Request

✓ Provide clear description of changes

✓ Link to related issue/ticket

✓ Request code review from peers (minimum 1 reviewer)

✓ For security changes: Request security team review

Step 7: Code Review (Reviewer Responsibility)

✓ Reviewer must check: See Section 6 for mandatory checklist

✓ Reviewer approves or requests changes

✓ If AI-generated code: Extra scrutiny required

Step 8: Automated CI Pipeline

✓ GitHub Actions runs all tests

✓ Security scanning (SAST, dependency check)

✓ Code coverage check (must maintain threshold)

✓ Build verification

Step 9: Merge (Only After CI Passes)

✓ Use "Squash and merge" for feature branches

✓ Only merge to main/develop with approval

✓ Delete branch after merging

Step 10: Deploy to Staging First

✓ NEVER deploy directly to production

✓ Test feature in staging environment

✓ Verify with product/QA team

✓ Monitor for errors in staging

Step 11: Deploy to Production

✓ Only authorized personnel can deploy (Tech Lead, DevOps)

✓ Use blue-green deployment or feature flags

✓ Monitor metrics for 30 minutes post-deploy

✓ Have rollback plan ready

4.2 Commits: Rules & Standards

Commit Message Format (Required): 
type(scope): subject (max 50 chars)

Body (optional, max 72 chars per line):
- Explain WHAT and WHY, not HOW
- Reference issue: Closes #123
- Add breaking changes note if applicable

Example:
feat(auth): add two-factor authentication

Add TOTP-based 2FA for user accounts.
- Users can enable 2FA in settings
- Backup codes generated on setup
- Login requires 6-digit code

Closes #456

Types (required):

  • feat - New feature
  • fix - Bug fix
  • docs - Documentation only
  • style - Code style (formatting, missing semicolons)
  • refactor - Code refactoring
  • perf - Performance improvement
  • test - Adding or updating tests
  • chore - Build, dependencies, tooling
  • security - Security fix
✅ GOOD Commit Message: 
fix(payment): prevent duplicate charges on retry

When payment API times out, retry logic was creating
duplicate charges. Now use idempotency key to prevent
duplicate processing.

Closes #789
❌ BAD Commit Message: 
Fixed bug
Updated code
Changed stuff
WIP

    6. Code Quality & Review Standards

    How to validate AI-generated code before it goes to production.

    6.1 Mandatory Code Review Checklist

    For ALL code (AI-generated or not):

    • Code passes linter without errors (npm run lint)
    • Code is formatted correctly (npm run format)
    • All tests pass locally (npm test)
    • No hardcoded credentials or secrets
    • Follows team naming conventions and patterns
    • Functions/components have clear purpose (name + comments)
    • Error handling is comprehensive (try/catch, error boundaries)
    • Performance: No obvious inefficiencies (N+1 queries, unnecessary renders)
    • Security: Input validation, output encoding, auth checks
    • Backward compatibility maintained (or breaking change documented)
    • Documentation updated (README, comments, ADR if applicable)

    EXTRA checks for AI-generated code:

    • Reviewer read and understood every line of code
    • Edge cases are handled (null, undefined, empty arrays, errors)
    • Error messages are helpful (not generic "Error occurred")
    • Logging is appropriate (no sensitive data logged)
    • Type safety verified (if using TypeScript)
    • Database queries analyzed for performance (EXPLAIN plan check)
    • Security review completed (OWASP Top 10 check)
    • Code matches our code standards (not just "works")
    • Claude's limitations acknowledged (AI isn't perfect)
    • At least one manual test performed (not just automated tests)

    6.2 Security Review Checklist

    OWASP Top 10 Quick Check:

    • No SQL injection: Use parameterized queries, never string concatenation
    • No broken authentication: Proper password hashing, token validation
    • No sensitive data exposure: Encrypt at rest, HTTPS in transit, no logging secrets
    • No XML external entities (XXE): If parsing XML, disable external entities
    • No broken access control: Authorization checks on all endpoints
    • No security misconfiguration: Proper security headers, CORS, CSRF tokens
    • No XSS: Output encoding, CSP headers, sanitize user input
    • No insecure deserialization: Don't deserialize untrusted data
    • No using components with known vulnerabilities: Update dependencies
    • No insufficient logging/monitoring: Log security events

    Specific Code Review Questions:

    • Is user input validated on both client AND server?
    • Are all database queries using parameterized queries?
    • Are passwords hashed (not stored in plain text)?
    • Are API endpoints protected with authentication/authorization?
    • Is sensitive data encrypted in transit (HTTPS)?
    • Are error messages generic (not leaking implementation details)?
    • Are rate limits in place to prevent brute force attacks?
    • Are CORS headers properly configured?
    • Is CSRF protection enabled for state-changing operations?
    • Are dependencies up to date (no known vulnerabilities)?

    6.3 What Reviewers Should Look For (AI Code Specific)

    Red Flag 1: Missing Error Handling 
    // Claude generated code without error handling
const response = await fetch('/api/users');
const data = await response.json();
// What if API returns 500? What if network fails?
// Claude often omits error handling!
    Red Flag 2: N+1 Query Problem 
    // Bad: Loops through users, queries database for each
users.forEach(user => {
  const posts = db.query('SELECT * FROM posts WHERE user_id = ?', user.id);
  // Now you have N+1 queries!
});

// Better: Load all data at once
const users = db.query('SELECT u.*, p.* FROM users u
                        LEFT JOIN posts p ON u.id = p.user_id');
    Red Flag 3: Insecure Password Handling 
    // BAD - Password stored in plain text
const password = req.body.password;
db.save({ email, password }); // NEVER do this!

// GOOD - Password hashed
const bcrypt = require('bcrypt');
const hashedPassword = await bcrypt.hash(password, 12);
db.save({ email, password: hashedPassword });
    Red Flag 4: Missing Input Validation 
    // BAD - No validation
app.post('/users', (req, res) => {
  db.insert(req.body); // What if email is invalid?
});

// GOOD - Validation before insert
app.post('/users', (req, res) => {
  const schema = Joi.object({
    email: Joi.string().email().required(),
    password: Joi.string().min(8).required()
  });

  const { error, value } = schema.validate(req.body);
  if (error) return res.status(400).json({ error });

  db.insert(value);
});
    Red Flag 5: Missing Authorization Check 
    // BAD - No authorization
app.delete('/users/:id', (req, res) => {
  db.delete('users', req.params.id); // Any user can delete any user!
});

// GOOD - Check authorization
app.delete('/users/:id', (req, res) => {
  const user = db.find('users', req.params.id);
  if (user.id !== req.user.id && req.user.role !== 'admin') {
    return res.status(403).json({ error: 'Unauthorized' });
  }
  db.delete('users', req.params.id);
});

    6.4 Performance Review Checklist

    • Database queries use indexes on frequently searched columns
    • No N+1 queries (use eager loading/joins)
    • API responses are paginated for large datasets
    • Caching used appropriately (Redis, HTTP cache headers)
    • Frontend: No unnecessary re-renders (React.memo, useMemo)
    • Frontend: No blocking operations on main thread
    • No inefficient loops or nested loops
    • Memory leaks prevented (cleanup in useEffect, event listener removal)
    • Large objects/arrays split into chunks/streams if needed
    • Build optimization verified (tree shaking, code splitting, minification)

    10.2 Secure Coding Checklist

    Before You Deploy, Check:

    • No secrets in code, comments, or git history
    • All user inputs validated (client AND server)
    • Database queries use parameterized statements (no string concat)
    • Output encoded to prevent XSS
    • Authentication required for protected endpoints
    • Authorization checked (user has permission)
    • HTTPS only (no HTTP)
    • CORS configured correctly (not * wildcard)
    • CSRF tokens on state-changing operations
    • Passwords hashed (bcrypt, scrypt, argon2)
    • Sensitive data encrypted (TLS in transit, encryption at rest)
    • Error messages don't leak implementation details
    • Rate limiting on authentication endpoints
    • Logging doesn't include sensitive data
    • Dependencies checked for vulnerabilities (npm audit, composer audit)
    • Security headers set (CSP, X-Frame-Options, etc.)
    • API versioning in place (/api/v1/...)
    • Documentation updated with security notes
    • Security review completed
    • Tests include security test cases
