1. THREAT MODEL: Real Attacks Against Claude Code Usage

Attack Vector 1: Secret Leakage

🎯 THE THREAT:

Developer accidentally shares API key with Claude. Key ends up in training data. Attacker finds key. Attacker uses key to access production systems, steal data, or create fraudulent charges.

Real Impact: $50k-$500k in fraudulent charges, data breach, customer notification, legal liability

🛡️ LAYERED DEFENSE (Defense in Depth):
  • Layer 1 - Prevention: Developers NEVER type actual secrets. Use placeholder values (sk_test_xxx). .env files auto-excluded from git (.gitignore configured).
  • Layer 2 - Detection: GitHub secret scanning enabled. Pre-commit hooks scan for patterns (AWS_KEY=, API_KEY=, password=). Local scanning on every commit.
  • Layer 3 - Monitoring: Git history scanned monthly with truffleHog. Any credentials found trigger automatic alert + rotation.
  • Layer 4 - Rotation: All secrets rotated quarterly regardless. If leaked, rotated within 1 hour + API call audit trail reviewed.
  • Layer 5 - Vault Storage: Production secrets ONLY in AWS Secrets Manager or HashiCorp Vault. Never in code/config/env vars on filesystem.

Attack Vector 2: Production Data Exposure

🎯 THE THREAT:

Developer shares production database dump with Claude to "optimize query". Contains 100k customer records. GDPR violation. Customer notification required. €20M fine.

Real Impact: GDPR/CCPA fines, customer lawsuit, reputation damage, bank account freeze

🛡️ LAYERED DEFENSE:
  • Layer 1 - Policy: EXPLICIT: "Never ask Claude about production data, customer data, PII, or production database details."
  • Layer 2 - Testing Environment: Developers get READ-ONLY access to anonymized test database only. Production database access requires VPN + MFA + audit log + approval.
  • Layer 3 - Detection: DLP (Data Loss Prevention) scans detect PII patterns in Claude conversations. Triggers automatic: block sharing + alert security team.
  • Layer 4 - Review: Random sample of 10 Claude conversations per month reviewed by security team for data leakage.
  • Layer 5 - Anonymization: If testing data needed, use anonymized copies: hashed IDs, fake names, synthetic emails, date shifting.

Attack Vector 3: Code Vulnerabilities Introduced by Claude

🎯 THE THREAT:

Claude generates authentication code missing password hashing. Developer trusts it. Code goes to production. Attacker dumps password database. All users compromised.

Real Impact: User credential compromise, account takeover, identity theft, regulatory violation

🛡️ LAYERED DEFENSE:
  • Layer 1 - Code Review (MANDATORY): ALL code reviewed by senior developer. AI code reviewed with EXTRA scrutiny (assume bugs). Checklist: OWASP Top 10, SQL injection, XSS, auth bypass, hardcoded secrets.
  • Layer 2 - Automated Scanning: SAST (Static Application Security Testing) on every PR: Snyk, SonarQube, Semgrep. Blocks merge if critical issues found.
  • Layer 3 - Dynamic Testing: Security tests in CI pipeline: dependency scanning, secrets detection, license compliance.
  • Layer 4 - Manual Testing: QA team functional + security testing. Tests edge cases Claude might miss (error handling, race conditions, boundary conditions).
  • Layer 5 - Staging Verification: ALL code deployed to staging FIRST. Monitored for 48 hours minimum before production. Error rate must be 0.

Attack Vector 4: Rogue Agents With High Permissions

🎯 THE THREAT:

Agent configured with database WRITE access to "optimize queries". Agent error corrupts 10,000 records. Incident response costs $100k. Data recovered from backup but 2-hour downtime.

Real Impact: Data corruption, downtime, incident costs, customer SLA breach

🛡️ LAYERED DEFENSE:
  • Layer 1 - Zero-Write Policy: Agents NEVER have write permission to ANY system. Period. Read-only to test data ONLY.
  • Layer 2 - Test Environment Only: Agents can only touch dev/staging systems. Production completely isolated (different VPC, different credentials, different network).
  • Layer 3 - Data Restrictions: Agent can only see anonymized test data. No real customer data, no real emails, no real transactions.
  • Layer 4 - Audit Logging: EVERY agent action logged: user, timestamp, agent name, action taken, data accessed, result. Immutable log (cannot be deleted).
  • Layer 5 - Automated Kill Switch: Agent has hard timeout (15 minutes max). If exceeds, automatically killed. Monthly review of all agent actions for anomalies.

2. PRACTICAL SECURITY HARDENING

Secret Management: Bulletproof Implementation

❌ INSECURE (Current State for Many Teams):

  • API keys in .env file in repository
  • Production credentials in application config
  • Hardcoded database passwords
  • No rotation of credentials
  • No audit of who accessed secrets

✅ BULLETPROOF (Hardened Implementation):

  • ALL secrets in AWS Secrets Manager or HashiCorp Vault (NOT environment variables)
  • Automatic rotation: 30 days for API keys, 90 days for database passwords
  • Every secret access logged: who, when, what secret, IP address
  • Staging secrets DIFFERENT from production (separate credentials)
  • Developers can NEVER see actual production secrets (masked in logs)
  • Git .gitignore includes: .env*, *.key, *.pem, secrets.*, config/database.yml
  • Pre-commit hook: scans for credential patterns BEFORE commit allowed
  • Monthly audit: GitHub secret scanning + custom script checking for leaked patterns

3. CRITICAL RULES (Non-Negotiable)

RULE 1: NO SECRETS, EVER

If you type an actual API key, password, or token → YOU violated policy → Access revoked immediately. No exceptions. No "I didn't know" defense.

RULE 2: NO PRODUCTION DATA

If you share customer data, PII, or production database info with Claude → Incident. Investigation. Potential termination. GDPR fine is the company's problem, not yours, but access will be revoked.

RULE 3: CODE REVIEW IS MANDATORY

All code must be reviewed by someone other than the author. If you commit without review → revert + mandatory meeting with tech lead. Second violation → access suspended.

RULE 4: STAGING FIRST, ALWAYS

Never deploy directly to production. Always stage first. If you do → access suspended for 2 weeks + mandatory incident postmortem + all future PRs require 2 senior reviews.

RULE 5: AGENTS = READ-ONLY TEST DATA ONLY

Agents never have write access. Agents never see production data. Agents never modify anything. Period. If you configure an agent with more permissions → access revoked + investigation.

4. INCIDENT RESPONSE (When Things Go Wrong)

Scenario: Secret Leaked

Timeline:

  • T+0 minutes: Developer realizes secret shared. Immediately tells manager and security team (email or Slack #security-incidents).
  • T+5 minutes: Security team starts incident. Assigns incident commander.
  • T+15 minutes: Secret rotated (API key disabled, new key generated, old key deleted).
  • T+30 minutes: Audit log checked: was secret accessed after sharing? (Check AWS Cloudtrail, API logs, database connection logs).
  • T+60 minutes: Investigation complete. Document: what leaked, when, who could access, was it used, impact assessment.
  • T+24 hours: Post-mortem: how did it happen, what prevents it next time, developer retraining, access status decision.

Scenario: Vulnerable Code Deployed

Timeline:

  • T+0 minutes: Security monitoring detects unusual activity (failed logins, data access anomaly, error rates spike). Auto-alert.
  • T+5 minutes: On-call engineer notified. Incident commander initiated.
  • T+15 minutes: Rollback decision made. If vulnerability confirmed → immediate rollback to previous version (blue-green switch).
  • T+30 minutes: Rollback complete. System restored. Monitoring confirms normal operation.
  • T+60 minutes: Root cause analysis: what vulnerability was missed in review? Why did SAST not catch it?
  • T+24 hours: Post-mortem + code review process improvement. Stricter rules for AI code going forward.

5. CONTINUOUS SECURITY OPERATIONS

Daily

  • Pre-commit hook scans for secrets (automated, blocks bad commits)
  • Error monitoring alerts on unusual patterns
  • Failed login attempts monitored (too many = incident)

Weekly

  • Security team reviews access logs (who accessed what, when)
  • Failed deployments analyzed (if multiple failures = pause deployments + investigate)
  • Code review audit (spot check 5 recent PRs for thoroughness)

Monthly

  • Git repository secret scan (truffleHog or GitGuardian)
  • Dependency vulnerability scan (Snyk, OWASP Dependency Check)
  • Random Claude conversation sample review (5-10 conversations)
  • Access log review (ensure least privilege maintained)
  • Credentials rotation (if not auto-rotated)

Quarterly

  • Full security audit of Claude Code usage
  • Agent action logs reviewed for anomalies
  • Incident trends analyzed (any patterns?)
  • Team security training (refresher on threats and defenses)
  • Penetration testing scenario (simulate secret leak, test response)

6. RED TEAM SCENARIOS (Test Your Defenses)

Scenario 1: What if developer shares AWS key by mistake?

  • Pre-commit hook should catch it (regex pattern for AWS keys)
  • If bypassed, GitHub secret scanning should flag it
  • If somehow missed, manual audit monthly would find it
  • Even if leaked, key is auto-rotated quarterly
  • Audit logs show if key was used (would be detected)

Scenario 2: What if agent tries to write to production database?

  • Agent credentials don't have write permissions (database-level enforcement)
  • Write attempt fails (permission denied error logged)
  • Error triggers alert (write attempt to prod = suspicious)
  • Incident investigation: why did agent try to write?
  • Agent access suspended pending investigation

Scenario 3: What if Claude-generated code bypasses auth check?

  • Code review should catch it (mandatory security checklist)
  • If missed in review, SAST scanning (SonarQube) should flag it
  • If somehow deployed, integration tests catch it (endpoint should return 401)
  • Staging deployment reveals error (48-hour monitoring before prod)
  • Incident postmortem: why was auth bypass not caught?

8. SUMMARY: Bulletproof Principles

Defense in Depth: Multiple layers. One layer fails, others still protect.
Zero Trust: Never assume security. Always verify. Assume compromise.
Least Privilege: Every user has minimum needed. Nothing more.
Audit Everything: If it's not logged, it didn't happen.
Detect Anomalies: Normal operations baseline. Any deviation = alert.
Respond Fast: When something goes wrong, minutes matter. Have playbook ready.
Learn & Improve: Every incident = postmortem = process improvement.
